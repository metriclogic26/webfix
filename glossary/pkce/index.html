PKCE (Proof Key for Code Exchange)

PKCE (Proof Key for Code Exchange, pronounced "pixy") is an OAuth 2.0 extension that prevents authorization code interception attacks. A client generates a random code verifier and sends its hash (code challenge) with the auth request. Only the original client can exchange the code because only it knows the verifier.

The attack PKCE prevents

Without PKCE, a malicious app on the same device can intercept the authorization code from the redirect URL (especially on mobile via custom URL schemes). It then exchanges the code for tokens before the legitimate app can.

PKCE makes stolen codes useless — the attacker doesn't have the code verifier needed to exchange them.

How PKCE works

  1. Client generates random code_verifier (43-128 chars)
  2. Client hashes it: code_challenge = BASE64URL(SHA256(code_verifier))
  3. Sends code_challenge and code_challenge_method=S256 with auth request
  4. After redirect, sends original code_verifier with token exchange request
  5. Server verifies: SHA256(code_verifier) must match stored challenge

Provider configuration

ProviderWhere to enable
Auth0Applications → Settings → Advanced → Grant Types: enable Authorization Code with PKCE
OktaApplications → Sign-On Policy → require PKCE
CognitoApp client → Enable PKCE-only auth flows
GoogleSupported by default for installed app clients

When to use PKCE

PKCE is required for all public clients — single-page apps, mobile apps, and desktop apps that cannot store a client secret. As of OAuth 2.1 (draft), PKCE is required for all authorization code flows, including confidential clients.
